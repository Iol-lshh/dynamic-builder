# 유스케이스 작성 가이드

앨리스터 코오번(Alistair Cockburn)의 유스케이스 원칙 기반. spec-analyze에서 참조한다.

## 핵심 원칙

**하나의 유스케이스 = 하나의 기본 흐름(Main Success Scenario)**

기본 흐름이 둘 이상이면 반드시 유스케이스를 분리한다. 유스케이스 내 분기는 **확장(Extensions)** 으로만 표현하며, 확장은 기본 흐름의 특정 스텝에서 파생되는 대안/예외 흐름이다.

## 분리 기준 체크리스트

아래 중 **하나라도** 다르면 별개의 유스케이스다:

| # | 기준 | 질문 |
|---|------|------|
| 1 | **기본 흐름** | 행위자가 목표를 달성하는 스텝 시퀀스가 동일한가? |
| 2 | **트리거** | 유스케이스를 시작하는 이벤트/요청이 동일한가? |
| 3 | **주 행위자** | 유스케이스를 개시하는 행위자가 동일한가? |
| 4 | **사전 조건** | 시작 전에 참이어야 하는 상태가 동일한가? |
| 5 | **사후 조건(성공 보장)** | 성공 시 시스템이 보장하는 상태가 동일한가? |
| 6 | **사용자 목표** | 행위자가 달성하려는 목표가 하나인가? |

## 분리 신호

다음 징후가 보이면 유스케이스를 분리하라:

- **"흐름 A", "흐름 B"** 같은 병렬 기본 흐름이 등장한다
- 기본 흐름 안에 **"또는", "혹은"** 으로 연결된 독립 시나리오가 있다
- **CRUD 오퍼레이션**을 하나로 묶고 있다 (Create/Read/Update/Delete 각각은 독립된 목적과 흐름을 가진다)
- 하나의 유스케이스에서 **사후 조건이 조건부**로 갈린다 ("A를 했으면 X 상태, B를 했으면 Y 상태")
- 기본 흐름의 **첫 번째 스텝부터 분기**한다 (확장은 기본 흐름 도중에 파생되는 것이지, 처음부터 갈리는 것이 아니다)

## 안티패턴

### 1. CRUD 합치기

```markdown
## 유스케이스: 소설 데이터 파이프라인 버퍼 관리    ← 잘못됨

### 기본 흐름
- 흐름 A (put): 버퍼에 데이터를 적재한다
- 흐름 B (getByFifo): 버퍼에서 FIFO로 꺼낸다
- 흐름 C (delete): 버퍼에서 데이터를 삭제한다
```

**왜 잘못인가:** put/get/delete는 트리거, 사전 조건, 사후 조건, 기본 흐름이 모두 다르다. "흐름 A/B/C"가 등장한 시점에서 이미 3개의 유스케이스다.

### 2. "또는"으로 연결된 목표

```markdown
## 유스케이스: 소설 색인 처리    ← 잘못됨

### 기본 흐름
1. 전체 색인 요청을 수신하거나 증분 색인 이벤트를 수신한다
2. ...
```

**왜 잘못인가:** 전체 색인과 증분 색인은 트리거(배치 스케줄러 vs Kafka 이벤트)와 기본 흐름이 다르다.

## 올바른 예시

위의 안티패턴을 올바르게 분리한 형태:

```markdown
## 유스케이스 1: 파이프라인 버퍼에 소설 데이터 적재

### 주 행위자: 배치 스케줄러
### 트리거: 전체 색인 배치 시작
### 사전 조건: 색인 대상 소설 데이터가 DB에 존재한다
### 성공 보장: 대상 데이터가 버퍼에 적재되어 있다

### 기본 흐름
1. 배치 스케줄러가 색인 대상 소설 데이터를 조회한다
2. 조회된 데이터를 문서로 변환한다
3. 변환된 문서를 버퍼에 적재한다

---

## 유스케이스 2: 파이프라인 버퍼에서 소설 데이터 소비

### 주 행위자: 색인 태스크릿
### 트리거: 색인 스텝 실행
### 사전 조건: 버퍼에 적재된 데이터가 존재한다
### 성공 보장: FIFO 순서로 데이터가 반환되고 버퍼에서 제거된다

### 기본 흐름
1. 색인 태스크릿이 버퍼에서 FIFO 순서로 데이터를 꺼낸다
2. 꺼낸 데이터를 검색엔진에 색인 요청한다

---

## 유스케이스 3: 파이프라인 버퍼 초기화

### 주 행위자: 배치 스케줄러
### 트리거: 색인 배치 완료 또는 실패 시 정리
### 사전 조건: 파이프라인 버퍼가 존재한다
### 성공 보장: 버퍼의 모든 데이터가 삭제되어 있다

### 기본 흐름
1. 배치 스케줄러가 버퍼 초기화를 요청한다
2. 버퍼의 모든 데이터를 삭제한다
```

## 기본 흐름 작성 규칙

**기본 흐름은 flat한 번호 리스트로만 작성한다 (들여쓰기/서브스텝 금지)**

- 기본 흐름 스텝 안에 서브스텝(2-1, 2-2 등)을 들여쓰기하지 않는다
- 스텝 내 세부 분기가 필요하면 **확장(Extensions)**으로 분리한다
- 스텝 내 변환 세부사항은 **기술 및 데이터 변형** 섹션에 기술한다

### 금지 형식

```markdown
### 기본 흐름
1. 클라이언트가 요청을 보낸다
2. Mapper가 요청을 변환한다
   - 2-1. 쿼리를 변환한다          ← 서브스텝 금지
   - 2-2. 필터를 변환한다          ← 서브스텝 금지
   - 2-3. 정렬을 변환한다          ← 서브스텝 금지
3. 서비스가 검색을 실행한다
```

**왜 금지인가:** 기본 흐름은 행위자의 목표 달성까지의 추상화된 행위 단계다. 변환의 세부 필드별 처리는 구현 디테일이지 행위 단계가 아니다. 세부사항은 기술 및 데이터 변형에, 분기는 확장에 표현한다.

### 올바른 형식

```markdown
### 기본 흐름
1. 클라이언트가 검색 요청을 전송한다
2. Mapper가 요청을 도메인 모델로 변환한다
3. 서비스가 검색엔진에 쿼리를 전달하고 결과를 수신한다

### 확장 (대안/예외 흐름)
- 2a. 쿼리가 비어있으면: 예외 발생
- 2b. 필터가 null이면: 기본 필터 적용

### 기술 및 데이터 변형
- 변환 대상: 쿼리, 필터, 정렬, 랭킹, 페이징, 요청자 정보
```

## 확장(Extensions) 포맷 규칙

**필수 형식: 코오번 확장 흐름 (bullet list)**

확장은 반드시 기본 흐름의 스텝 번호를 참조하는 bullet list로 작성한다. 테이블 형식은 사용하지 않는다.

### 올바른 형식

```markdown
### 확장 (대안/예외 흐름)
- 2a. [조건]: [대안 흐름]
  - 2a1. [세부 사항]
  - 2a2. [세부 사항]
- 2b. [조건]: [예외 흐름]
- 3a. [조건]: [예외 흐름]
```

### 금지 형식

```markdown
### 확장 (대안/예외 흐름)
| 조건 | 흐름 |            ← 테이블 형식 금지
|---|---|
| 5a. RateLimiter 초과 | ... |
```

**왜 금지인가:** 테이블은 기본 흐름 스텝과의 연결 관계, 확장 간의 중첩(2a → 2a1)을 표현할 수 없다. 코오번 유스케이스의 확장은 "기본 흐름 N번 스텝에서 [조건]이면" 이라는 구조적 의미를 가지므로, 이를 정확히 표현하는 bullet list가 유일한 올바른 형식이다.

## 확장과 별개의 유스케이스 구분

| 구분 | 확장(Extension) | 별개의 유스케이스 |
|------|-----------------|-------------------|
| 기본 흐름 | 동일 (특정 스텝에서 분기) | 다름 |
| 트리거 | 동일 | 다름 |
| 사후 조건 | 동일 (또는 실패로 종료) | 다름 |
| 예시 | "2a. 데이터가 없으면: 빈 결과 반환" | put vs get vs delete |
