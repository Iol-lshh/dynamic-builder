# 컨텍스트 격리 원칙

LLM 세션에서 컨텍스트 오염을 방지하고, 올바른 근거 기반의 산출물을 보장하기 위한 원칙.
모든 워크플로우(Spec, TDD, Coverage)의 실행 모델은 이 원칙을 기반으로 설계한다.

## 문제: 컨텍스트 오염

LLM 세션의 컨텍스트는 **누적적**이고 **자기강화적**이다.

1. **오염 전파** — 잘못된 근거(hallucination, 부정확한 가정, 오류가 있는 중간 추론)가 컨텍스트에 한번 들어가면, 후속 추론이 이를 사실로 참조한다.
2. **자기 일관성 편향** — LLM은 이전 출력과 모순되는 결론을 내리기 어렵다. 틀린 방향이어도 일관성을 유지하려는 경향이 있다.
3. **체인 구조의 증폭** — 각 단계가 이전 단계의 출력을 입력으로 받는 구조에서, 초기 오염이 파이프라인 전체로 전파된다.

**핵심 문제**: 오염된 컨텍스트 안에서 "수정"을 시도하는 행위 자체가 오염된 근거 위에서 이루어진다. 따라서 오염 발생 후 세션 내 교정은 근본적으로 한계가 있다.

## 원칙

### 1. 단일 책임 서브에이전트 (SRP Isolation)

**각 작업 단계는 독립된 서브에이전트에서 수행한다.**

- 메인 세션은 오케스트레이션만 담당하고, 실제 분석/생성 작업은 서브에이전트가 수행
- 서브에이전트는 단일 책임(SRP)만 가짐 — 한 에이전트가 여러 성격의 작업을 혼합하지 않음
- 서브에이전트의 컨텍스트는 작업 완료 시 폐기됨 — 오염이 발생해도 다른 단계로 전파되지 않음

```
[메인 세션: 오케스트레이션만]
    ├→ [서브에이전트 A: 분석] → 산출물 파일
    ├→ [서브에이전트 B: 생성] → 산출물 파일
    └→ [서브에이전트 C: 검증] → 산출물 파일
```

**왜 중요한가**: 메인 세션에서 직접 작업하면, 중간 추론 과정이 모두 컨텍스트에 쌓여서 후속 판단에 영향을 준다. 서브에이전트로 격리하면 중간 추론은 서브에이전트와 함께 폐기되고, 최종 산출물만 파일로 남는다.

### 2. 산출물의 외부화 (Externalized Artifacts)

**작업 결과는 메인 세션의 컨텍스트가 아닌 파일로 문서화한다.**

산출물에는 반드시 두 가지가 포함되어야 한다:

| 구성 요소 | 목적 |
|---|---|
| **결정의 근거** | 왜 이 결론에 도달했는지의 추론 과정. 다음 단계에서 잘못된 근거를 식별/교정 가능 |
| **산출물 본문** | 분석 결과, 명세, 코드 등 실제 작업 결과물 |

**왜 중요한가**:
- 메인 세션의 컨텍스트가 중간 추론으로 오염되지 않음
- 근거가 명시적이므로 검증 가능 — 암묵적 가정이 숨어들지 않음
- 필요할 때만 파일을 읽어서 참조 (lazy loading) — 컨텍스트 효율성

**안티패턴**: 서브에이전트의 결과를 메인 세션 컨텍스트에 요약해서 넣는 것. 요약 과정에서 왜곡이 발생할 수 있고, 이후 메인 세션의 모든 추론에 영향을 미친다.

### 3. 유저 게이트키핑 (Human In The Loop)

**각 단계 사이에 사용자가 산출물을 확인하고 다음 단계의 수행과 입력을 결정한다.**

```
서브에이전트 작업 → 산출물 파일 → [사용자 확인] → 다음 단계 결정
                                      │
                                      ├→ 승인: 다음 단계 진행
                                      ├→ 수정 요청: 같은 단계 재실행
                                      └→ 거부: 입력을 수정하여 재실행
```

**왜 중요한가**:
- 오염된 근거가 다음 단계로 넘어가는 것을 차단하는 유일한 안전장치
- LLM이 스스로 자기 오류를 감지하는 것은 구조적으로 한계가 있음 (자기 일관성 편향)
- 사용자만이 도메인 지식을 기반으로 근거의 타당성을 판단할 수 있음

## 적용

이 원칙은 프로젝트의 모든 워크플로우에 공통 적용된다:

| 워크플로우 | 적용 |
|---|---|
| **Spec Workflow** | 3단계(analyze → plan → review) 각각 서브에이전트, plan은 병렬 2개로 편향 방지, review에서 사용자 선택 |
| **TDD Workflow** | 3단계(test → implement → quality-review) 각각 서브에이전트, 매 단계 HITL |
| **Coverage Workflow** | 4단계(analyze → plan → test → review) 각각 서브에이전트, 매 단계 HITL |

## 메인 세션의 역할

메인 세션은 다음만 수행한다:

1. **오케스트레이션** — 워크플로우 단계의 순서 관리
2. **서브에이전트 실행** — 적절한 프롬프트와 입력으로 서브에이전트 호출
3. **산출물 전달** — 사용자에게 산출물 위치를 알리고 확인 요청
4. **사용자 결정 수신** — 다음 단계의 수행 여부와 입력 결정

메인 세션은 직접 분석, 생성, 판단을 하지 않는다. 이를 통해 메인 세션의 컨텍스트를 깨끗하게 유지한다.
