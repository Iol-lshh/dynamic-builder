# 답변 분석 가이드 (answer-analysis-guide)

## 목적

사용자 답변과 Phase 1의 3개 analyst 산출물을 교차 분석하여, 리포트 작성에 필요한 ASIS/TOBE 요소를 추출한다.

---

## 입력

| 소스 | 내용 |
|------|------|
| report-question-lifecycle 산출물 | task-lifecycle 관점 분석 결과 및 질문 |
| report-question-usecase 산출물 | usecase-design 관점 분석 결과 및 질문 |
| report-question-tradeoff 산출물 | technical-tradeoff 관점 분석 결과 및 질문 |
| report-question-synthesis 산출물 | 통합 질문 목록 |
| 사용자 답변 | HITL #1에서 수집된 답변 (런타임 주입) |

---

## 분석 절차

### 1단계: 답변-질문 정합성 검증

| 검증 항목 | 기준 |
|-----------|------|
| 응답 완전성 | 각 질문에 대해 답변이 존재하는가 |
| 응답 관련성 | 답변이 질문의 의도에 부합하는가 |
| 누락 답변 | 답변되지 않은 질문 목록 식별 |

**누락 답변 처리**: 답변이 없는 질문은 "미확인" 표시하고, 해당 ASIS/TOBE 요소를 "추정"으로 태그한다. 추정 근거는 analyst 산출물에서 도출한다.

### 2단계: ASIS 요소 추출

| 추출 대상 | 소스 |
|-----------|------|
| 현재 상태 | 사용자 답변 중 현 상황 기술 부분 |
| 현재 문제점 | 사용자 답변 중 불만/제약 언급 부분 |
| 배경 맥락 | analyst 산출물의 사실 기술 부분 |

### 3단계: TOBE 요소 추출

| 추출 대상 | 소스 |
|-----------|------|
| 목표 상태 | 사용자 답변 중 기대/희망 기술 부분 |
| 변경 범위 | 사용자 답변 + analyst 산출물의 범위 분석 |
| 성공 기준 | 사용자 답변 중 완료 조건 언급 부분 |

### 4단계: 교차 검증

- analyst 산출물에서 도출한 ASIS/TOBE 가설을 사용자 답변으로 확인 또는 기각한다
- analyst 간 상충하는 분석이 있으면 사용자 답변을 기준으로 조율한다
- 사용자 답변에서 새로 발견된 요소(analyst가 미탐지)를 별도 표시한다

## 산출물 형식

```markdown
## 답변-질문 정합성

| 질문 ID | 질문 요약 | 답변 상태 | 비고 |
|---------|-----------|-----------|------|
| Q1 | ... | 응답됨/미확인 | ... |

## ASIS 요소

| # | 요소 | 근거 (답변/분석) | 확신도 |
|---|------|-----------------|--------|
| 1 | ... | 답변: "..." / 분석: lifecycle-Q2 | 확인됨/추정 |

## TOBE 요소

| # | 요소 | 근거 (답변/분석) | 확신도 |
|---|------|-----------------|--------|
| 1 | ... | 답변: "..." / 분석: usecase-Q1 | 확인됨/추정 |

## 교차 검증 결과

| 항목 | analyst 가설 | 사용자 답변 | 판정 |
|------|-------------|------------|------|
| ... | ... | ... | 확인/기각/신규 |
```

---

## 금지 사항

- 사용자 답변에 없는 내용을 ASIS/TOBE로 확정하지 않는다 (추정 태그 필수)
- analyst 산출물의 분석을 답변 없이 리포트 요소로 승격하지 않는다
- 답변의 의도를 자의적으로 해석하지 않는다
