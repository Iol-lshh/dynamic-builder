# Scope Reduction

범위·복잡도 수준에서 사고한다. 현재 설계나 구현에서 무엇이 핵심이고 무엇이 과잉인지 식별하여, 최소 실현 가능한 범위를 정의한다.

## 렌즈

- **출발점**: 복잡도가 진전을 막는 설계 또는 구현
- **기술 수준**: "무엇이 불필요한가(what is excess)" — 지금 꼭 필요한 것만 남기면 무엇인가
- **코드 참조**: 제거 가능 요소와 의존성 식별 용도
- **산출물 수준**: 제거 목록(CUT), 최소 해법(MINIMUM), 후속 분리 계획(DEFER)

## 판단 기준

- 지금 당장 없어도 되는 요소가 있는가 (CUT)
- 핵심 요구사항만으로 구성된 최소 해법은 무엇인가 (MINIMUM)
- 지금 필요하지 않지만 나중에 필요한 것을 분리할 수 있는가 (DEFER)
- 단순화 후에도 원래 목적을 달성할 수 있는가
- 복잡도의 원천이 요구사항인가, 구현 방식인가

## 이 관점이 적합한 작업

- 설계·구현의 복잡도 과부하 상황
- 범위 축소 후에도 진전 없을 때
- MVP 범위 재설정
- 기능 우선순위 재조정
