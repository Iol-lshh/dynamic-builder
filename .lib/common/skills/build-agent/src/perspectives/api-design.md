# API Design

API 계약 수준에서 사고한다. 인터페이스의 명세, 호출자와 제공자 간의 계약, 통신 프로토콜에 집중한다.

## 렌즈

- **출발점**: 호출자(consumer)가 필요로 하는 계약
- **기술 수준**: "어떤 계약으로(contract)" — 입력/출력/오류가 무엇인가
- **코드 참조**: 계약이 올바르게 구현되었는지 확인 용도
- **산출물 수준**: 엔드포인트, 요청/응답 스키마, 오류 코드, 통신 프로토콜

## 판단 기준

- API 계약이 호출자 관점에서 명확하고 일관적인가
- 요청/응답 스키마가 필요충분한가 (과도하지도 부족하지도 않은가)
- 오류 응답이 호출자가 대응할 수 있는 수준으로 구체적인가
- 하위 호환성을 고려했는가
- 통신 프로토콜(REST, gRPC, protobuf 등)의 제약을 반영했는가

## 이 관점이 적합한 작업

- API 엔드포인트 설계 및 리뷰
- 시스템 간 통신 인터페이스 정의
- 요청/응답 DTO 설계
- API 문서화
