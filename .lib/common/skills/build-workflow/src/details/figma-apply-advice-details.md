# figma-apply-advice-details.md

스텝: apply-advice
에이전트: scope-reduction-reconciler

---

## 목적

통합 백로그에 대해 스코프 어드바이스를 생성하고, 그 결과를 즉시 반영하여 최종 백로그를 산출한다. retry 시에는 평가 피드백도 함께 반영한다.

## 입력

- `backlog-draft.md`: merge-analyses의 산출물 (백로그 초안)
- `evaluation-backlog.md`: evaluate-backlog의 산출물 (retry 시에만 존재)

### evaluate-backlog null 허용 조건부 로직

- **초회 실행**: `evaluation-backlog.md`가 존재하지 않는다. 이 경우 스코프 어드바이스 생성 + 반영만 수행한다.
- **retry 실행**: `evaluation-backlog.md`가 존재한다. 평가 피드백의 감점 사유를 확인하고, 해당 항목을 우선 보정한 뒤 스코프 어드바이스를 재생성 + 반영한다.

## Phase A: 스코프 어드바이스 생성

각 백로그 항목에 대해 아래 3가지 판정 중 하나를 부여한다:

### CUT (제거)

- MVP에 불필요하거나 핵심 사용자 시나리오와 무관한 항목
- 판정 근거를 1문장으로 명시한다

### MINIMUM (최소 구현)

- 필수이나 초기 범위를 축소할 수 있는 항목
- 최소 구현 범위를 구체적으로 명시한다 (예: "3가지 상태 중 default만 구현")

### DEFER (후순위)

- 필요하나 현 스프린트에서 구현하지 않아도 되는 항목
- 연기 가능한 이유를 1문장으로 명시한다

### 판정 기준

- 판정 없음 = 현행 유지 (AS-IS). 모든 항목에 판정을 강제하지 않는다.
- CUT + MINIMUM + DEFER 합계가 전체 항목의 30-50%가 되도록 목표한다
- 50%를 초과하면 범위 과다 축소, 30% 미만이면 축소 효과 부족

## Phase B: 어드바이스 반영

Phase A의 판정을 백로그에 즉시 반영한다:

1. **CUT 항목**: 백로그에서 제거한다. 제거된 항목 목록을 부록에 기록한다.
2. **MINIMUM 항목**: 설명과 구현 힌트를 최소 구현 범위로 수정한다.
3. **DEFER 항목**: 백로그에 유지하되 "[DEFER]" 태그를 붙인다.
4. **판정 없는 항목**: 그대로 유지한다.

## Phase C: 평가 피드백 반영 (retry 시에만)

`evaluation-backlog.md`가 존재하면:

1. 감점 사유 목록을 확인한다
2. 각 감점 사유에 해당하는 백로그 항목을 식별한다
3. 해당 항목의 설명, 구현 힌트, 스코프 판정을 보정한다
4. 보정 내용을 "[피드백 반영]" 태그로 표시한다

## 산출물 형식

```markdown
# 최종 백로그

## BL-{번호}: {항목 제목}

- **출처**: uiux / business / 통합
- **유형**: {유형}
- **스코프**: AS-IS / MINIMUM / DEFER
- **설명**: {보정된 설명}
- **근거**: {참조 노드}
- **구현 힌트**: {보정된 구현 힌트}

---

## 부록: 제거된 항목 (CUT)

| 번호 | 제목 | CUT 근거 |
|------|------|----------|
| BL-xxx | ... | ... |

## 부록: 스코프 어드바이스 요약

- 전체 항목: N개
- AS-IS: N개 / MINIMUM: N개 / DEFER: N개 / CUT: N개
- 축소 비율: XX%
```
