# 질문 도출 가이드 (question-generation-guide)

## 목적

입력 데이터(작업한 내용, 확인받아야 할 내용, 회의에서 들은 내용)를 자신의 관점(perspective)으로 분석하여, 보고서 작성에 필요한 질문을 도출한다.

---

## 입력 데이터 유형

| 유형 | 설명 | 분석 초점 |
|------|------|-----------|
| 작업한 내용 | 실제 수행한 업무, 코드 변경, 문서 작성 등 | ASIS(현재 상태)와 변경 사항 파악 |
| 확인받아야 할 내용 | 의사결정 필요 항목, 승인 대기 사항 | TOBE(목표 상태) 관련 불확실성 |
| 회의에서 들은 내용 | 논의된 사항, 결정사항, 미결 사항 | 맥락, 제약조건, 이해관계자 의도 |

---

## 관점별 탐색 프레임

각 analyst는 자신의 perspective에 정의된 렌즈와 판단 기준을 적용하여 질문을 도출한다.

### 공통 탐색 축

| 축 | 질문 방향 |
|----|-----------|
| ASIS 파악 | 현재 상태는 무엇인가? 왜 이 상태인가? |
| TOBE 파악 | 목표 상태는 무엇인가? 왜 이 목표인가? |
| GAP 식별 | ASIS와 TOBE 사이에 무엇이 빠져 있는가? |
| 동기 확인 | 왜 이 작업/변경/결정이 필요한가? |
| 범위 확인 | 무엇이 포함되고 무엇이 제외되는가? |
| 방법 확인 | 어떻게 달성할 계획인가? |

---

## 질문 작성 규칙

### 질문 포맷

```
[배경] — 이 질문을 도출한 입력 데이터의 구체적 부분
[질문] — 확인이 필요한 사항
[의도] — 이 질문의 답변이 보고서의 어떤 섹션에 기여하는지
```

### 작성 원칙

1. **구체성**: "전반적인 상황은?" 같은 모호한 질문 금지. 입력 데이터의 특정 부분을 지시하여 질문
2. **비중복**: 동일 관점 내에서 같은 정보를 묻는 질문을 반복하지 않음
3. **답변 가능성**: 사용자가 답변할 수 있는 수준의 질문. 사용자가 모를 가능성이 높은 기술적 세부사항은 제외
4. **ASIS/TOBE 연결**: 모든 질문은 ASIS 파악 또는 TOBE 파악에 기여해야 함
5. **개수 제한**: 관점당 3~7개. 핵심 질문에 집중

---

## 산출물 형식

```markdown
## {perspective-name} 관점 질문

### 질문 1
- **배경**: {입력 데이터에서 발견한 사실}
- **질문**: {확인이 필요한 사항}
- **의도**: {ASIS/TOBE 중 어디에 기여하는지}

### 질문 2
...
```

---

## 금지 사항

- 입력 데이터에 없는 내용을 가정하여 질문하지 않는다
- 다른 관점의 전문 영역에 속하는 질문을 도출하지 않는다
- 보고서 작성이 아닌 작업 수행을 위한 질문을 도출하지 않는다
